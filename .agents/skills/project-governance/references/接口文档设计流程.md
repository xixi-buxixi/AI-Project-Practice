# 接口文档设计流程通用规范

> 适用场景：在需求文档冻结后、代码构建启动前，将需求阶段产出的逻辑实体、状态机和事件流，落地为当前项目类型对应的物理接口或交互契约。本文档不默认项目一定是 Web API，适用于 Web/API、CLI、桌面端、浏览器插件、数据任务、自动化脚本、AI Agent、移动端、游戏和混合型项目。

接口文档不是让 AI 自由发挥生成的接口清单，而是连接 ReqOps 需求资产与 CodeOps 代码构建的物理契约。它必须从冻结后的需求资产包推导而来，并能被审查、测试、Hook 和模块 Agent 持续追踪。

> **定位说明**：本文是 `AI项目构建流程.md` 的 ContractOps 阶段细化规范，不单独作为全流程入口。项目执行级别、角色裁剪、Hook 输出和清理规则以主构建流程为准。

> **入口补充**：接口文档生成前必须完成需求阶段的 Grill Gate。若当前工具支持 `/grill-me-doc` 或等价能力，AI 应先基于冻结需求资产追问契约模糊点；若工具不支持斜杠命令，则使用 `AI指导.md` 中的兼容提示词。

## 1. 核心原则

- 接口文档必须晚于需求冻结，早于模块编码。
- 接口文档生成前必须先通过 Grill Gate / Grill-With-Docs，确认冻结需求中的术语、扩展点、边界和验收标准已经足够明确。
- 接口文档描述的是物理契约，不只包含 HTTP API，也包含命令、消息、事件、工具调用、数据输入输出、权限和失败策略。
- AI 不能脱离 PRD、术语表、逻辑实体、事件流和模块边界凭空生成接口。
- 需求阶段只生成逻辑接口草案，不能提前锁死 HTTP Method、Path、数据库字段名、SDK 签名或命令参数。
- 物理契约必须覆盖 PRD 中的核心路径、异常路径、状态机流转和验收标准。
- 物理契约不得引入 Out of Scope 中明确不做的功能。
- 所有字段、状态值、错误码和事件名必须与术语表和逻辑接口保持一致。
- 写操作、资金相关操作、外部副作用操作和高风险操作必须定义幂等、鉴权、审计和失败恢复策略。
- 物理契约必须包含追踪映射，能反向映射到 REQ、AC、逻辑实体和业务事件。
- PRD 中标记为 E2 及以上的扩展点，物理契约必须定义统一扩展接口、选择规则、能力声明或注册方式；不得只给每个成员追加一套散落的接口说明。
- 当契约需要支持多平台、多渠道、多供应商、多内容类型、多支付方式、多工具或多格式时，必须说明新增一个扩展成员时需要新增或修改哪些契约文件，以及哪些核心契约禁止修改。
- 物理契约必须通过所选级别要求的只读审查后，才能移交 CodeOps；高风险或 Enterprise 项目必须额外通过 RegressionAgent 和 Auditor。

## 2. 总体流程

```text
1. 读取冻结后的 ReqOps 资产包
   ↓
2. 执行准入检查，确认无 P0/P1 模糊项，并完成 Grill-With-Docs 追问
   ↓
3. 根据 Lite / Standard / Enterprise 选择裁剪级别
   ↓
4. 根据项目类型选择物理契约模板
   ↓
5. 编写 docs/contracts/overview.md
   ↓
6. 按模块编写 docs/contracts/modules/<module-name>.md
   ↓
7. 补充扩展点契约、错误矩阵、权限边界、幂等、限流、重试和追踪映射
   ↓
8. 执行反向覆盖校验和反分支堆积校验
   ↓
9. 只读视角检查 Out of Scope 回归；高风险或 Enterprise 启用 RegressionAgent
   ↓
10. 按裁剪级别执行审查；高风险或 Enterprise 启用 Auditor 异构盲审
   ↓
11. 修订并最多循环 3 轮
   ↓
12. Human Owner 确认 PASS / CONDITIONAL PASS
   ↓
13. OrchestratorAgent 挂载契约并移交 ModuleAgent
```

## 3. 输入锚定

在生成任何物理契约之前，OrchestratorAgent 或 ContractAgent 必须读取冻结后的 ReqOps 资产包。

### 3.1 必读输入

| 输入文件 | 用途 |
|---|---|
| `docs/requirements/overview.md` | 总览 PRD，提供目标、流程、状态机、验收标准和非功能要求 |
| `docs/requirements/glossary.md` 或 `docs/requirements/overview.md` 中的 Glossary 章节 | 统一术语、字段命名、业务口径和状态枚举；Standard 合并版没有独立 glossary 时必须读取兜底章节 |
| `docs/requirements/out-of-scope.md` | 防止接口层引入需求外能力 |
| `docs/requirements/modules/<module-name>.md` | 模块职责、输入输出、验收标准和边界 |
| `docs/interfaces/logical-model.md` | 逻辑实体、关键属性、关系和约束 |
| `docs/interfaces/events.md` | 业务事件、触发条件、生产者、消费者和状态变化 |
| `MODULE_MAP.md` | 模块归属、依赖关系、交付物和物理契约路径 |
| `docs/decisions/decisions-and-pending-log.md` | 已裁决取舍、延期项和条件放行项 |
| `docs/traceability/traceability-source.yaml` 或 Markdown annotations | REQ、AC、模块、测试和代码映射来源 |
| PRD 中的扩展性设计要求 | 扩展点、扩展强度、推荐模式、新增成员允许修改和禁止修改的边界 |

### 3.2 准入检查

存在以下情况时，不允许生成物理契约：

- 项目类型未明确。
- PRD、模块需求、逻辑实体或事件流尚未冻结。
- P0/P1 级 `待确认`、`TBD` 或业务冲突仍未解决。
- 逻辑实体或事件流无法承载 PRD 的状态机。
- 模块边界不清，无法判断接口的 Producer 和 Consumer。
- Out of Scope 未记录。
- 术语表中的关键业务词未冻结或仍存在多义定义。
- PRD 已标记 E2 及以上扩展点，但没有说明扩展成员、变化频率、推荐模式或新增成员边界。

## 4. 裁剪级别

接口文档需要跟随项目规模裁剪，裁剪只能减少文档数量，不能取消需求边界、异常路径、安全检查和验证结果。

### 4.1 Lite

适用于单页应用、小工具、自动化脚本、个人原型或一次性数据处理。

推荐做法：

- 可以将接口契约合并进 `REQUIREMENTS.md` 的独立章节。
- 若接口超过 3 个，建议独立创建 `CONTRACTS.md`。
- 至少记录入口、输入、输出、错误、权限或配置、验收映射。
- 不强制维护 `docs/contracts/modules/`。

### 4.2 Standard

适用于多数小系统、内部工具、轻量商业 MVP、单端 Web/App 或服务端 API。

Standard 级默认只保留“契约生成者 + 实现者 + 只读审查者”的闭环。只有当物理契约出现范围漂移、高风险写操作、安全隐私、资金资损或连续审查冲突时，才额外启用 `RegressionAgent` / `Auditor`。

推荐结构：

```text
project-root/
├─ docs/
│  ├─ requirements/
│  │  └─ overview.md
│  ├─ architecture.md
│  └─ contracts/
│     ├─ overview.md
│     └─ modules/
│        └─ <module-name>.md
├─ MODULE_MAP.md
└─ agent_prompts/
```

Web/API 项目可以额外生成 OpenAPI 文件，但必须保持与 `docs/contracts/` 的映射关系：

```text
docs/
├─ api/
│  └─ openapi.yaml
└─ contracts/
   ├─ overview.md
   └─ modules/
      └─ <module-name>.md
```

### 4.3 Enterprise

适用于金融、安全、隐私、合规、多团队、长期维护或高风险项目。

建议保留完整结构：

```text
project-root/
├─ docs/
│  ├─ requirements/
│  │  ├─ constitution.md
│  │  ├─ glossary.md
│  │  ├─ overview.md
│  │  ├─ out-of-scope.md
│  │  └─ modules/
│  │     └─ <module-name>.md
│  ├─ interfaces/
│  │  ├─ logical-model.md
│  │  └─ events.md
│  ├─ contracts/
│  │  ├─ overview.md
│  │  └─ modules/
│  │     └─ <module-name>.md
│  ├─ traceability/
│  ├─ decisions/
│  └─ reviews/
├─ MODULE_MAP.md
└─ agent_prompts/
```

Enterprise 项目应优先使用结构化 Schema，例如 OpenAPI、JSON Schema、YAML Config、AsyncAPI、SQL Schema 或工具定义 Schema。

## 5. 项目类型与物理契约映射

| 项目类型 | 物理契约必须包含 | 推荐格式 |
|---|---|---|
| Web / SaaS / API 服务 | HTTP Method、Path、Header、鉴权、Request Body、Response Body、错误码、限流、幂等、WebSocket 或异步事件 | `docs/contracts/` + OpenAPI 3.x |
| CLI / 桌面工具 | Command、Subcommand、Arguments、Flags、配置文件、Exit Code、stdout、stderr、文件输入输出 | Markdown Help Manual + JSON/YAML Schema |
| 浏览器插件 | Manifest Permissions、Content Script、Background、Message Passing、Runtime Event、宿主页交互限制 | JSON Schema + Event Matrix |
| 移动 App / 小程序 | 端能力调用、设备权限、本地缓存、离线同步、推送消息、版本兼容 | Markdown + JSON Schema |
| 数据任务 / ETL / 报表 | Input Source Schema、Output Destination Schema、调度规则、重试、幂等、失败重跑、数据口径 | SQL/JSON Schema + YAML Config |
| 自动化脚本 / 运维工具 | 运行入口、参数、权限、目标资源、回滚策略、日志格式、失败退出规则 | Markdown + Config Schema |
| AI Agent / 工作流 | Tool Definition、Tool Input/Output、Context Memory、Human-in-the-loop、循环上限、Fallback | JSON Schema + Markdown |
| 游戏 / 互动体验 | 输入事件、状态同步、存档结构、关卡数据、实时消息、失败恢复 | JSON Schema + Event Contract |

## 5.1 扩展点与物理契约映射

PRD 中每个 E2 及以上扩展点必须映射到一种或多种物理契约形态。

| 扩展强度 | 契约要求 | 典型文件 |
|---|---|---|
| E1 轻量枚举 | 枚举值、字段约束、兼容策略 | `docs/contracts/overview.md` |
| E2 策略/适配器 | 统一输入输出 Schema、能力差异、选择规则、默认实现、错误映射 | `docs/contracts/extensions/<extension-point>.md` |
| E3 插件化/注册表 | 注册表格式、能力声明、版本兼容、生命周期、权限边界、审核或启停规则 | `docs/contracts/extensions/<extension-point>.md` + `extension-registry.schema.json` |
| E4 规则引擎/工作流 | 规则 Schema、执行上下文、优先级、冲突处理、审计、回滚 | `docs/contracts/rules/<rule-domain>.md` |

示例：多平台内容发布不应为每个平台各自散落定义一组互不相关的发布接口，而应先定义 `PublishPlatformAdapter` 的统一契约，再为每个平台记录能力差异、字段映射、鉴权、限流、错误恢复和发布状态同步。

## 7. 编写规则

### 7.1 命名规则

- HTTP 路由、CLI 命令、配置键和文件名默认使用小写短横线或项目既有风格。
- JSON 字段默认使用项目既有风格，未明确时使用 `camelCase`。
- 状态枚举和错误码使用大写蛇形命名，例如 `PENDING`、`SUCCESS`、`TASK_DUPLICATE_SUBMIT`。
- 字段含义必须能在术语表或逻辑实体中找到来源。
- 不允许同一业务概念在不同契约中使用多个名字。
- 扩展点名称、适配器名称、策略名称和注册表键必须稳定，不得直接使用临时 UI 文案或第三方营销名称作为核心契约标识。

### 7.2 输入输出规则

每个输入和输出字段必须记录：

- 字段名。
- 类型。
- 是否必填。
- 取值范围、长度、格式或枚举。
- 业务含义。
- 来源术语或逻辑实体。
- 默认值和空值行为。

不允许只写示例 JSON 而不写字段约束。

### 7.3 扩展契约规则

当某个业务维度达到 E2 及以上时，ContractAgent 必须优先定义扩展契约，而不是复制粘贴多套相似接口。

扩展契约至少包含：

- 扩展点名称和业务目的。
- 统一输入 Schema 与输出 Schema。
- 扩展成员的能力声明，例如是否支持草稿、定时、图片、视频、回调、撤销、重试、批量处理。
- 扩展成员选择规则，例如按平台、内容类型、租户、配置、用户选择或系统策略。
- 默认实现或不支持时的失败行为。
- 错误码映射和恢复策略。
- 鉴权、密钥来源和权限边界。
- 限流、重试、幂等和审计规则。
- 新增扩展成员时允许新增或修改的文件。
- 新增扩展成员时禁止修改的核心契约、核心流程或状态机。

推荐模板：

```md
# Extension Contract: <ExtensionPoint>

## 1. Purpose

## 2. Strength

E2 / E3 / E4

## 3. Unified Capability Interface

| Capability | Required | Meaning | Unsupported Behavior |
|---|---:|---|---|
| <capability> | yes/no | <说明> | <失败或降级策略> |

## 4. Unified Input Schema

| Field | Type | Required | Constraint | Source Term |
|---|---|---:|---|---|

## 5. Unified Output Schema

| Field | Type | Required | Meaning | Source Term |
|---|---|---:|---|---|

## 6. Member Registry

| Member | Capability Declaration | Auth Requirement | Rate Limit | Error Mapping | Status |
|---|---|---|---|---|---|

## 7. Selection Rule

## 8. Add New Member Procedure

- Add:
- Update:
- Must Not Modify:

## 9. Acceptance Mapping
```

### 7.4 状态和事件规则

每个会改变业务状态的契约必须说明：

- 调用前允许的状态。
- 成功后进入的状态。
- 失败后进入的状态。
- 触发的业务事件。
- 事件的 Producer 和 Consumer。
- 是否允许重复触发。

### 7.5 错误和异常规则

不允许只写 Happy Path。每个契约必须包含 Error Matrix。

错误矩阵至少包含：

- 认证失败。
- 权限不足。
- 参数校验失败。
- 资源不存在。
- 状态冲突。
- 重复提交。
- 外部依赖失败。
- 超时。
- 限流。
- 存储、网络或系统资源不足。

示例：

```md
| Error Code | Physical Status | Meaning | Trigger | Recovery |
|---|---:|---|---|---|
| AUTH_TOKEN_EXPIRED | 401 | Token 已过期 | 登录凭证超时 | 前端重新登录或刷新 Token |
| TASK_DUPLICATE_SUBMIT | 409 | 重复提交任务 | 幂等键命中已有任务 | 返回原任务状态 |
| MEDIA_STORE_FULL | 507 | 存储空间不足 | 媒体写入失败 | 提示用户清理空间并记录告警 |
```

### 7.6 权限和隐私规则

每个契约必须说明：

- 调用者身份。
- 允许的角色。
- 数据隔离方式。
- 是否读取或返回敏感字段。
- 是否写入审计日志。
- 是否依赖 `.env` 中的配置。
- 是否会调用外部服务。

禁止在契约文档中写真实密钥、真实 Token、真实密码或真实连接串。

### 7.7 幂等、限流和重试规则

以下契约必须定义幂等策略：

- 创建、修改、删除类操作。
- 资金、库存、额度、积分、权益相关操作。
- 会调用外部系统的操作。
- 会产生文件、消息、任务或不可逆副作用的操作。

幂等策略应记录：

- 幂等键名称。
- 幂等键来源。
- 幂等键有效期。
- 重复请求返回策略。
- 并发冲突处理方式。

限流和重试策略应记录：

- 限流维度。
- 限流阈值。
- 超限错误码。
- 消费端是否可重试。
- 最大重试次数。
- 退避策略。
- 最终失败后的兜底行为。

## 8. Web/API 特殊规则

Web/API 项目的物理契约可以沉淀为 OpenAPI，但 Markdown 契约仍应保留业务解释、状态流转、错误恢复和追踪映射。

### 8.1 HTTP 契约必须包含

- Method。
- Path。
- Path Params。
- Query Params。
- Headers。
- Authentication。
- Request Body。
- Response Body。
- Error Matrix。
- Rate Limit。
- Idempotency。
- Event 或 WebSocket 推送关系。

### 8.2 WebSocket 容错契约

如果项目使用 WebSocket、SSE 或其他实时推送通道，必须明确：

- 实时通道只作为增量通知通道，不能作为唯一事实源。
- 前端初始化时必须通过 HTTP 或等价查询接口拉取全量状态。
- 断线重连后必须重新同步最新状态。
- 消息必须包含可排序或可去重的字段，例如 `eventId`、`version` 或 `occurredAt`。
- 消费端必须能处理重复、乱序、丢失和延迟消息。

示例：

```md
WebSocket 仅作为状态流的增量通知通道。前端初始化、断线重连或检测到事件版本缺口时，必须调用 `/api/session/init` 或等价接口同步全量最新状态。
```

## 9. AI Agent 契约特殊规则

AI Agent 项目的物理契约必须约束模型可调用的工具和上下文边界，不能只描述自然语言意图。

每个 Tool Contract 必须包含：

- Tool Name。
- Tool Purpose。
- Input JSON Schema。
- Output JSON Schema。
- Allowed Caller。
- Required Human Confirmation。
- Side Effects。
- Failure Modes。
- Retry Limit。
- Memory Read/Write Rules。
- Permission Boundary。
- Audit Log。

涉及删除、发送、支付、发布、外部调用、批量修改或不可逆动作时，必须设置 Human-in-the-loop 确认点。

## 10. 数据任务契约特殊规则

数据任务、ETL、报表和自动化脚本必须明确：

- 输入数据源。
- 输入 Schema。
- 输出位置。
- 输出 Schema。
- 调度方式。
- 分区或时间窗口。
- 增量处理规则。
- 重跑规则。
- 幂等规则。
- 脏数据处理。
- 失败告警。
- 数据口径来源。

指标口径必须回链到术语表或 PRD 中的数据统计定义。

## 11. 反向覆盖校验

物理契约编写完成后，ContractAgent 必须执行反向覆盖校验。

### 11.1 覆盖 PRD

检查每个核心功能是否至少有一个物理契约承载：

| PRD Item | Required Contract | Actual Contract | Status |
|---|---|---|---|
| <REQ> | <需要的交互> | <Contract ID> | covered / missing |

### 11.2 覆盖验收标准

检查每个 AC 是否能转成测试：

| AC | Contract | Testable By | Gap |
|---|---|---|---|
| <AC-001> | <Contract ID> | unit / integration / e2e / manual | <缺口> |

### 11.3 覆盖状态机

检查每个状态变化是否有触发入口和失败兜底：

| State Transition | Trigger Contract | Success Event | Failure Handling | Status |
|---|---|---|---|---|

### 11.4 Out of Scope 回归扫描

RegressionAgent 必须读取 `out-of-scope.md`，逐条扫描物理契约是否出现范围漂移。

发现以下情况时必须阻塞：

- 契约开放了 Out of Scope 中明确不做的能力。
- 契约通过通用参数绕过了范围限制。
- 契约新增了 PRD 未定义的用户角色、状态、权限或外部依赖。
- 契约让模块承担了 MODULE_MAP 外的职责。

### 11.5 扩展性与反分支堆积校验

ContractAgent 或只读审查者必须检查 PRD 中的扩展点是否被契约正确承载。

| Extension Point | Strength | Required Contract | Actual Contract | Add Member Boundary | Status |
|---|---|---|---|---|---|
| <扩展点> | E2/E3/E4 | <应有契约> | <实际契约> | <新增成员边界> | covered / missing / risky |

阻塞规则：

- PRD 标记为 E2 及以上，但物理契约没有统一输入输出 Schema、选择规则或能力声明时，视为 P1。
- PRD 标记为 E3 及以上，但物理契约没有注册表、插件声明或成员生命周期规则时，视为 P1。
- 同一扩展维度出现三套及以上相似契约，却没有抽象统一扩展点时，视为 P1。
- 契约要求新增扩展成员时修改核心状态机、核心发布流程、核心支付流程或核心调度流程，且没有 Decision Log 说明时，视为 P1。
- 允许通过少量条件判断承载 E1 扩展，但必须说明不会升级为持续增长的成员集合。
- 契约设计与代码实现中提倡使用设计模式（如策略模式、状态模式或查表法）重构代码，消除硬编码和冗余的分支逻辑，提高内聚性。

## 12. Auditor 异构盲审

物理契约不能孤立审查。Auditor 必须只读审查，不修改文件。

### 12.1 审查输入

Auditor 必须同时读取：

- 最终 PRD。
- 术语表。
- Out of Scope。
- 模块需求。
- 逻辑实体。
- 事件流。
- 物理契约。
- MODULE_MAP。
- 决策日志。
- 追踪矩阵或 Markdown annotations。

### 12.2 评分维度

| 维度 | 说明 | 通过阈值 |
|---|---|---:|
| contract_consistency_score | 物理契约是否与术语表、逻辑实体、事件流和模块边界一致 | >= 8 |
| business_closure_score | 物理契约是否能完整承载 PRD 状态机、核心路径和异常路径 | >= 8 |
| interface_completeness_score | 输入、输出、错误、权限、幂等、限流、重试是否完整 | >= 8 |
| extensibility_contract_score | PRD 中的扩展点是否有统一契约、注册方式、新增成员边界和反分支堆积约束 | >= 8 |
| acceptance_testability_score | 契约是否能直接转成测试用例和验收脚本 | >= 8 |
| risk_control_score | 安全、隐私、资损、越权、配置和失败兜底是否充分 | >= 7 |

评分证据规则：

- 任一维度低于阈值时，必须在 Gap List 中给出对应证据。
- 证据必须定位到 PRD、术语表、逻辑实体、事件流、物理契约或 `MODULE_MAP.md` 的具体路径和章节。
- 不能用“感觉不完整”“建议加强”“可能有风险”作为阻塞理由。
- 如果无法给出证据，低分只能标记为观察项，不能触发 REJECT。

### 12.3 Gap List

Auditor 给出 REJECT 或 CONDITIONAL PASS 时，必须输出 Gap List：

```md
## Gap List

| Gap | Score Impact | Blocking Level | Evidence | Required Fix | Owner |
|---|---|---|---|---|---|
| <缺口> | <扣分维度> | P0/P1/P2/P3 | <证据> | <需要补齐什么> | <负责人> |
```

### 12.4 三态结论

| 结论 | 条件 | 后续动作 |
|---|---|---|
| PASS | 所有分数达到阈值，且无 P0/P1 | 冻结物理契约，移交 CodeOps |
| CONDITIONAL PASS | 无 P0/P1，但存在资源、工期或可接受风险 | Human Owner 写入 Decision Log 后条件放行 |
| REJECT | 任意关键分数低于 6，或存在 P0/P1 | 打回修订，禁止进入 CodeOps |

P0 安全、隐私、资损、主流程闭环缺失和不可验收问题不能被条件放行。

## 13. 移交 CodeOps

物理契约通过审查后，由 OrchestratorAgent 执行移交。

### 13.1 更新 `MODULE_MAP.md`

每个模块必须记录物理契约路径：

| Module | Requirement Doc | Logical Interface Doc | Physical Contract Doc | Dependencies | Acceptance Criteria |
|---|---|---|---|---|---|
| <Module> | `docs/requirements/modules/<module>.md` | `docs/interfaces/logical-model.md` | `docs/contracts/modules/<module>.md` | <依赖> | <AC> |

### 13.2 更新 ModuleAgent 上下文

每个 ModuleAgent 的提示词必须要求读取：

- `AI_AGENT_CONTRACT.md`。
- `MODULE_MAP.md`。
- 对应模块需求文档。
- `docs/interfaces/logical-model.md`。
- `docs/interfaces/events.md`。
- 对应模块物理契约文档。
- 相关审查问题和决策日志。

### 13.3 Hook 对齐

构建阶段 Hook 应检查：

- 物理契约文档是否存在。
- 物理契约是否比 ModuleAgent 提示词更新。
- 代码或测试是否包含 `maps: REQ-xxx`。
- 高风险接口是否包含 `contracts: <Contract ID>` 或等价映射。
- 测试是否覆盖核心 Contract ID。
- 契约变更后是否重新触发所选级别要求的只读审查；高风险或 Enterprise 需重新触发 RegressionAgent 和 Auditor。

Hook 输出必须遵循主构建流程中的 JSON 契约，至少报告：

- 缺失的 Contract ID。
- 未覆盖的 AC。
- 代码或测试中缺失的 `maps:` / `contracts:` 注解。
- 需要人工裁决的 P0/P1 契约冲突。

建议代码注释：

```js
// maps: REQ-001
// contracts: API-TASK-001
```

```python

# contracts: CLI-EXPORT-001
```

## 14. 变更治理

需求冻结后，如果 PRD、逻辑实体、事件流或物理契约发生变化，必须走变更流程。

流程：

1. 暂停受影响的 ModuleAgent。
2. 记录变更原因和影响范围。
3. 更新 PRD、逻辑接口或物理契约。
4. 更新 `MODULE_MAP.md`。
5. 更新追踪映射。
6. 只读视角执行 Out of Scope 回归检查；高风险或 Enterprise 启用 RegressionAgent。
7. 高风险或 Enterprise 项目由 Auditor 对变更部分重新盲审。
8. OrchestratorAgent 同步 ModuleAgent 上下文。
9. 重新运行受影响模块的测试和 Hook。

禁止只改代码、不改契约。也禁止只改契约、不同步 Agent 上下文和测试。

## 15. 最小使用方式

如果不执行完整流程，至少完成以下步骤：

1. 确认需求文档已冻结，且无 P0/P1 模糊项。
2. 明确项目类型，不默认 Web API。
3. 读取 PRD、术语表、Out of Scope、逻辑实体、事件流和 MODULE_MAP。
4. 编写一个最小物理契约文档，包含入口、输入、输出、错误、权限、状态变化和追踪映射。
5. 如果 PRD 存在 E2 及以上扩展点，补充统一扩展契约、能力声明、选择规则和新增成员边界。
6. 检查物理契约是否覆盖核心验收标准。
7. 检查是否引入 Out of Scope。
8. 检查是否通过散落接口或条件分支继续堆积同一扩展维度。
9. 由另一个只读视角进行审查，并要求问题包含证据位置。
10. 若涉及安全、隐私、资金、合规或不可逆副作用，再启用 Auditor 盲审。
11. 通过后再移交代码构建。

## 16. 接口文档冻结退出标准

接口文档可以进入 CodeOps，需要同时满足：

- 项目类型和契约形式已明确。
- 物理契约从冻结后的 ReqOps 资产包推导而来。
- Grill Gate 已完成，且接口层的模糊术语、扩展点和边界已被明确记录。
- 每个模块都有对应的物理契约，或在裁剪级别中有明确合并说明。
- 每个契约都有输入、输出、错误、权限、状态变化和追踪映射。
- E2 及以上扩展点都有对应扩展契约、能力声明、选择规则和新增成员边界。
- 同一扩展维度没有通过三套及以上散落契约或条件分支继续堆积；如需例外，已写入 Decision Log。
- 写操作和高风险操作已定义幂等策略。
- 实时通道已定义断线重连和全量状态同步策略。
- 物理契约没有引入 Out of Scope。
- 只读审查无 P0/P1 回归问题；若启用 RegressionAgent，则其报告无 P0/P1。
- 若启用 Auditor，则 Auditor 无 P0/P1 契约问题。
- 若启用 Auditor，关键评分达到阈值，或 Human Owner 明确记录 CONDITIONAL PASS。
- OrchestratorAgent 已同步 MODULE_MAP 和 ModuleAgent 上下文。

## 17. 术语对照
