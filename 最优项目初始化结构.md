# 最优项目初始化结构

> **定位说明**：本文是 `AI项目构建流程.md` 的初始化结构备忘，用于快速搭建目录和文件职责。完整流程、裁剪规则、Hook 输出、清理策略与审查规则以主构建流程为准。

> **入口补充**：初始化项目结构前，AI 必须先读取 `AI指导.md` 和 `AI项目构建流程.md`，确认 Grill Gate、Goal Gate、需求资产与接口契约均已满足当前阶段准入条件。

> 目标：把 AI 协作项目从“人类愿景文档”收敛为“可执行协议”。 
> 核心原则：稳定导航和动态状态分离；只读必要上下文；任务状态只有一个来源；模块边界明确；扩展点有固定承载位置；长历史默认归档不读。

## 1. 推荐目录结构

```text
project-root/
├── AI_AGENT_CONTRACT.md        # AI 协作契约与全局执行底线规则（原 AGENTS.md）
├── REQUIREMENT_CONTRACT.md     # 需求阶段薄契约：冻结资产、准入条件和阻塞条件
├── agent.md                    # 项目稳定导航（全局上下文、模块边界、启动规则）
├── MODULE_MAP.md               # 模块路由表（连接需求、接口契约与负责 Agent）
├── README.md
├── .env.example
├── .gitignore
├── tasks.md                    # 全局交付级任务看板（总 Agent 维护摘要）
│
├── agent_prompts/              # 统一的 Agent 提示词目录
│   ├── OrchestratorAgent/      # 总 Agent（流水线调度器）静态初始化提示词
│   │   └── init.md
│   └── templates/              # 子 Agent 提示词动态模板
│       ├── executor.md         # 执行子 Agent 模板
│       └── reviewer.md         # 审查子 Agent 模板
│
├── docs/                       # 静态与冻结文档资产（Standard 级别合并）
│   ├── requirements/
│   │   └── overview.md         # 合并 PRD、核心原则、术语表与边界限制
│   ├── architecture.md         # 合并模块划分、逻辑实体模型与事件流
│   ├── contracts/
│   │   ├── overview.md         # 合并当前项目类型对应的物理接口或交互契约
│   │   ├── extensions/         # E2+ 扩展点契约，如平台/渠道/供应商/格式/工具
│   │   │   └── <extension-point>.md
│   │   └── modules/
│   │       └── <module-name>.md # 细分模块物理契约
│   ├── decisions/
│   │   └── decisions-and-pending-log.md # 记录决策、延期与 Pending 问题
│   └── reviews/                # 质量审查输出目录
│       └── review-round-xxx.md
│
├── hooks/                      # 强制自动化 Hook 工具目录
│   ├── preflight.py            # 前置检查脚本（验证依赖、状态与模块静态规则）
│   └── finalize.py             # 后置清理校验脚本（校验语法、Linter、白名单临时产物清理）
│
├── .agent/                     # 本地 Agent 状态目录，必须加入 .gitignore
│   ├── state/
│   │   └── cycles.json         # 轮次、失败分类与模块锁状态
│   └── quarantine/             # 未识别未追踪文件隔离区
│
└── src/                        # 业务源码目录
    └── <module_name>/
        ├── agents.md           # 模块静态规则（可修改文件范围、模块API边界、依赖与特定约束）
        ├── tasks.md            # 模块动态看板（执行子 Agent 必须在此更新工作总结与自测命令）
        ├── extensions/         # 本模块扩展实现：策略、适配器、插件、注册表
        │   ├── registry.*      # 扩展成员注册表或导出入口
        │   └── <member>.*      # 单个扩展成员实现
        └── ... (业务代码与局部测试)
```

## 2. 文件职责边界

### `AI_AGENT_CONTRACT.md`

真正给编码 Agent 读取的入口规则。只放全局强制执行规则，例如：

- 必须先读取根 `agent.md` 和全局任务看板 `tasks.md`。
- 必须根据任务范围读取对应模块下的 `agents.md` 和 `tasks.md`。
- 必须直接写入目标文件，不能只输出 diff、补丁文本或“请手动修改”的说明。
- 必须遵守需求和契约中的扩展强度：E2 及以上扩展点优先新增策略、适配器、插件或注册项，禁止把持续增长的业务差异堆成 `if/else`。
- 必须在进入初始化前完成 Grill Gate 和 Goal Gate，确认当前阶段只是在搭建结构，不是在实现业务。
- 必须在开发结束后运行后置 Hook。Hook 只能删除白名单临时产物；未识别的未追踪文件必须报告或隔离，不能直接删除。
- 修改后必须运行可用验证命令，并把结果写入对应模块的 `tasks.md`。

### `agent.md`

项目稳定导航。只放不频繁变化的信息：

- 项目目标和当前阶段。
- 模块清单和模块边界。
- 全局扩展点清单、强度和对应模块归属。
- 总览 Agent 与模块 Agent 的启动方式。
- 关键文档索引。
- 跨模块协作规则。

不得在根 `agent.md` 维护动态任务看板。

### `MODULE_MAP.md`

连接需求资产与物理代码的路由表：

- 列出所有模块划分。
- 定义各模块对应的需求文档路径、物理契约路径以及负责的 Agent。
- 标明模块间的依赖关系和交付物验收标准。
- 标明各扩展点归属模块、扩展强度、注册表路径、允许新增文件和禁止修改的核心流程。

推荐增加扩展列：

| Module | Extension Points | Strength | Registry Path | Add Member By | Must Not Modify |
|---|---|---|---|---|---|
| <module> | <platform/channel/provider> | E2/E3/E4 | <path> | <新增策略/适配器/插件> | <核心流程> |

### `agent_prompts/OrchestratorAgent/init.md`

总 Agent（流水线调度器）的角色职责与流程规则：

- 规定总 Agent 的调度逻辑与决策范围（不亲自审查代码，只运行 Hook 并阅读审查报告）。
- 规定如何根据任务要求动态提取验收指标并生成验收清单。
- 规定如何把需求文档中的扩展点强度注入模块 Agent，并在发现分支堆积时暂停实现、回到契约或决策日志。

### `src/<module_name>/agents.md`

模块静态规则。只放该模块的稳定设计与边界信息：

- 模块目标、职责范围、外部依赖与不负责的范围。
- 允许修改的文件路径与受限修改范围。
- 接口规范及特定 Hook 的运行方式。
- 本模块负责的扩展点、扩展强度、实现模式、注册表路径、新增成员流程和禁止修改的核心文件。

### `src/<module_name>/tasks.md`

模块局部动态看板。执行子 Agent 进度的唯一事实源：

- 记录该模块的当前任务目标、质量等级、完成状态（todo | in_progress | blocked | done）。
- 记录已完成总结、技术决策与详细自测日志。

## 3. 动态混合校验与防爆机制

校验流水线的深度与严格程度，根据任务派发时设定的**满意度目标**动态切换：

### 档位 1：一般 (Average / Prototype) ── 轻量验证通道
- 子 Agent 汇报完成后，总 Agent 执行 `hooks/finalize.py --lint` 和至少一个 smoke test。
- 若轻量验证通过，总 Agent 在全局看板标记为 Done。不实例化审查 Agent。

### 档位 2：合格 (Qualified / MVP) ── 自动化首检通道
- 子 Agent 汇报完成后，总 Agent 在本地静默执行 `hooks/finalize.py` 进行首检与用完即删清理。
- 总 Agent **仅读取测试日志的总结文本**，判定通过后直接在全局看板更新为 Done。不实例化审查 Agent。

### 档位 3：优秀 (Great / Production) ── 独立红蓝审计通道
- 本地运行 `hooks/finalize.py` 自动化首检通过后，总 Agent 默认读取结构化报告；若证据不足或出现高风险问题，可以定向阅读相关源码片段。
- 总 Agent 实例化一个独立的**【审查子 Agent】**，注入验收清单和待审代码目录。
- 审查子 Agent 运行深度边界测试，输出详细审查报告至 `docs/reviews/review-round-xxx.md` 并销毁。
- 总 Agent **只读取结构化审查报告**做出通过或打回的判定。

## 4. `tasks.md` 标准格式 (模块局部看板)

```markdown
# Module Task State: <module_name>

## Current
- Task ID: TASK-XXXX
- Goal: <任务目标描述>
- Target Level: Great | Qualified | Average
- Status: todo | in_progress | blocked | done
- Last Updated: YYYY-MM-DD HH:MM:SS

## Done Summary
*   [修改或增加的具体功能描述]
*   [变更的文件清单]

## Blocked (若有阻塞)
*   Blocker: <阻塞原因>
*   Needed Input: <需要的反馈或输入信息>

## Technical Decisions
*   [开发过程中形成的架构或设计决策]

## Verification
- Command: <验证命令，如 npm run test:login>
- Result: pass | fail | not_run
- Log Summary: <测试通过的简要控制台输出>
```

## 5. 最小读取协议

### 总览 Agent 默认读取

1. `/agent.md`
2. `/tasks.md` (全局看板)
3. `/agent_prompts/OrchestratorAgent/init.md`

### 模块执行 Agent 默认读取

1. `/AI_AGENT_CONTRACT.md`
2. `/src/<module_name>/agents.md`
3. `/src/<module_name>/tasks.md`
4. 与本模块相关的 `/docs/contracts/extensions/<extension-point>.md`（若任务涉及 E2+ 扩展点）

## 6. 模块激活指令模板

总 Agent 实例化执行子 Agent时注入：

```markdown
# Activate Module Agent: <module_name>

## Required Reads
1. `/AI_AGENT_CONTRACT.md`
2. `/src/<module_name>/agents.md`
3. `/src/<module_name>/tasks.md`

## Task Context
- Goal: <具体任务目标>
- Allowed Paths: <只允许修改的路径>
- Target Level: Great | Qualified | Average
- Extension Points:
  - Name: <扩展点名称>
  - Strength: E0 | E1 | E2 | E3 | E4
  - Required Pattern: enum | strategy | adapter | factory | registry | plugin | rule-engine | workflow | event-driven
  - Registry Path: <注册表或能力声明路径>
  - Add Member By: <新增策略/适配器/插件/配置>
  - Must Not Modify: <核心流程或状态机路径>
- Auto-derived Check-list:
  1. [验收指标 1]
  2. [验收指标 2]

## Action Required
- 开发完成后，你必须在 `/src/<module_name>/tasks.md` 写入工作总结与自测命令。
- 如果任务涉及扩展点，必须优先新增独立策略、适配器、插件或注册项；除非 `Extension Points` 标明 E0/E1 或 Decision Log 有例外，不得在核心流程中追加并列条件判断。
- 在接受审查并由总 Agent 确认通过前，你必须保持活跃状态（不自行销毁会话）。
```

## 7. 压缩和归档规则

模块任务状态不得整体覆盖压缩。必须分区处理：

- `Current`：当前任务，永不压缩。
- `Technical Decisions`：关键技术决策，只追加或精炼，不删除未失效约定。
- `Blocked`：阻塞信息在解除前不得压缩。
- `Done Summary`：已完成历史可以压缩成摘要。
- 详细历史日志移动到模块下的 `archive/` 目录中，默认不读取。

## 8. 初始化执行顺序

1. 创建根 `AI_AGENT_CONTRACT.md`，写入全局强制执行规则。
2. 创建根 `REQUIREMENT_CONTRACT.md`，写入需求冻结资产、进入接口/构建阶段的准入条件和阻塞条件。
3. 创建必要的需求、架构、物理契约、扩展契约和决策文档，确保 `docs/decisions/decisions-and-pending-log.md` 是唯一决策日志路径。
4. 创建根 `agent.md`，写入项目导航、模块边界和启动规则。
5. 创建根 `MODULE_MAP.md`，划分模块并指定对应的物理路径、扩展点归属、注册表路径和禁止修改的核心流程。
6. 创建 `agent_prompts/OrchestratorAgent/init.md` 注入总 Agent 的调度规则和扩展性治理规则。
7. 在每个模块的 `src/<module_name>/` 下创建 `agents.md`（静态规则）和 `tasks.md`（动态看板）。
8. 对涉及 E2+ 扩展点的模块，创建 `src/<module_name>/extensions/` 及最小注册入口。
9. 在全局 `tasks.md` 中写入第一批任务清单。
10. 按照依赖关系和模块激活指令启动具体模块工作。
11. 每次完成、阻塞或验证后，由总 Agent 或子 Agent 更新对应 `tasks.md`。
12. 在初始化结束时，确认 Grill Gate 已完成、Goal Gate 已冻结本轮结构目标，且没有把业务代码偷偷塞进初始化步骤。

## 9. 验收清单

- [ ] 根 `agent.md` 只包含稳定导航，不包含动态任务看板。
- [ ] 所有动态任务状态分别存在于全局 `tasks.md` 和各模块的 `src/<module_name>/tasks.md` 中。
- [ ] 模块的静态规则和边界均已放在 `src/<module_name>/agents.md` 中。
- [ ] 初始化前已通过 Grill Gate 和 Goal Gate，且本轮目标仅限于结构搭建。
- [ ] E2 及以上扩展点均已在 `MODULE_MAP.md` 标明归属模块、注册表路径、新增成员方式和禁止修改的核心流程。
- [ ] 需要扩展的模块已存在 `docs/contracts/extensions/<extension-point>.md` 或等价合并章节。
- [ ] 涉及扩展点的模块已预留 `src/<module_name>/extensions/` 或等价策略/适配器目录。
- [ ] 模块激活指令包含扩展强度、推荐模式和反分支堆积约束。
- [ ] 校验体系能够根据任务的满意度等级（优秀/合格/一般）进行动态深度切换。
- [ ] 后置 Hook 包含安全清理机制，只删除白名单临时产物，其他未追踪文件进入报告或隔离区。
- [ ] 执行子 Agent 在接受审查并由总 Agent 确认通过前保持存活，不提前销毁。
- [ ] 跨模块修改前后都更新总览任务状态。
- [ ] 每次完成任务前均有可追溯的 Hook 验证或审计报告记录。
- [ ] 同一扩展维度没有在核心流程中出现三处及以上并列 `if/else` 或 `switch/case`；如有例外，已记录到决策日志。
