# AI 项目启动与协作指导

> 目的：作为所有 AI 工具进入本项目流程的统一入口。无论编辑器是否支持 `/grill-me`、`/grill-me-doc` 或 `/goal`，AI 都必须按本文完成需求澄清、文档摄入、目标锁定和阶段准入检查。

## 1. 使用顺序

AI 接到新项目、新阶段或重大变更任务时，必须按以下顺序执行：

```text
1. 读取本文件 `AI指导.md`
   ↓
2. 读取 `AI项目构建流程.md`
   ↓
3. 按阶段读取：
   - 需求阶段：`需求文档设计流程.md`
   - 接口阶段：`接口文档设计流程.md`
   - 初始化阶段：`最优项目初始化结构.md`
   ↓
4. 执行 Grill Gate：追问模糊需求、术语冲突、扩展点、边界、验收标准
   ↓
5. 执行 Goal Gate：锁定当前阶段目标、完成条件、验证方式和阻塞条件
   ↓
6. 按阶段产出需求文档、接口文档、初始化结构或代码实现
```

禁止跳过前 5 步直接生成代码。

## 2. Grill Gate

Grill Gate 的目标是防止 AI 在模糊上下文中直接生成 PRD、接口文档或代码。

### 2.1 何时必须执行

- 用户只给出初始想法、草稿、口头描述或半成品需求时。
- 需求文档中存在 `待确认`、`TBD`、冲突、模糊词或未定义术语时。
- 项目涉及多平台、多渠道、多供应商、多内容类型、多支付方式、多格式、多 AI 工具或多业务策略时。
- AI 准备进入接口设计、项目初始化或代码构建前。
- 发生需求变更、范围争议、接口冲突或扩展性风险时。

### 2.2 支持斜杠命令时

如果当前工具支持 slash command，优先使用：

```text
/grill-me
```

用于普通需求澄清。

```text
/grill-me-doc
```

用于带文档摄入的需求、接口、术语和架构追问。如果工具中实际命令名是 `/grill-with-docs`，则使用 `/grill-with-docs`。

### 2.3 不支持斜杠命令时的兼容提示词

```md
请进入 Grill-Me 模式。

你要围绕我给出的项目想法、需求文档或阶段计划进行连续追问，直到目标、范围、边界、扩展点、验收标准和风险都清楚。

规则：
1. 一次只问一个问题。
2. 每个问题都给出你的推荐答案和理由。
3. 如果问题能通过读取当前项目文档或代码回答，你必须先读取，不要问我。
4. 发现模糊词、冲突、隐藏假设、范围漂移、扩展性风险时，必须指出。
5. 发现多平台、多渠道、多供应商、多内容类型、多支付方式、多格式、多 AI 工具或多策略时，必须输出扩展点清单，并建议 E0-E4 扩展强度。
6. 在没有完成追问前，不要生成最终 PRD、接口文档或代码。
7. 每轮输出：已明确内容 / 待确认问题 / 决策记录 / 下一步建议。
```

### 2.4 带文档摄入的兼容提示词

```md
请进入 Grill-With-Docs 模式。

先读取并主动摄入这些文档：
- AI指导.md
- AI项目构建流程.md
- 需求文档设计流程.md
- 接口文档设计流程.md
- 最优项目初始化结构.md
- docs/requirements/glossary.md（如果存在）
- docs/decisions/decisions-and-pending-log.md（如果存在）

然后对当前项目想法、需求文档或接口文档进行一问一答式追问：
1. 一次只问一个问题，并给出推荐答案。
2. 发现术语冲突时，指出旧定义和新含义的差异。
3. 发现模糊术语时，提出一个精确的标准术语。
4. 发现架构或扩展性决策时，记录到待决策清单。
5. 发现扩展点时，必须判断 E0-E4 强度，并说明推荐模式：枚举、策略、工厂、适配器、插件、注册表、规则引擎或事件驱动。
6. 只有当术语、边界、扩展点、验收标准都清楚后，才生成当前阶段文档草案。
```

## 3. Goal Gate

Goal Gate 的目标是防止 AI 在长任务中遗忘目标、跳阶段或在没有验证的情况下宣布完成。

### 3.1 何时使用

- 已完成 Grill Gate，当前阶段目标明确时。
- 需要 AI 连续完成一组可验证任务时。
- 需要跨多轮推进需求文档、接口文档、初始化结构或模块构建时。

不建议在需求模糊时直接使用 `/goal 构建整个项目`。

### 3.2 支持 `/goal` 时

按阶段设置目标，不要一次性把整个项目塞进一个目标。

推荐阶段目标：

```text
/goal 根据用户初始构想和《需求文档设计流程.md》生成需求文档草案，完成 Grill Gate，列出待确认问题、扩展点清单、E0-E4 强度、验收标准和是否可进入接口设计的判断。
```

```text
/goal 根据冻结需求资产和《接口文档设计流程.md》生成接口文档草案，补齐 E2+ 扩展契约、错误矩阵、权限边界、追踪映射和接口冻结退出检查。
```

```text
/goal 根据冻结需求与接口资产、主构建流程和最优初始化结构，生成项目初始化结构与模块 Agent 上下文，确保扩展点注册表和反分支堆积约束已落地。
```

```text
/goal 实现 <module-name> 模块，只修改允许路径，遵守模块契约和扩展点模式，运行验证并更新 tasks.md。
```

### 3.3 不支持 `/goal` 时的兼容提示词

```md
请把下面内容当作一个持久目标执行，即使当前编辑器不支持 /goal，也要按 Goal Mode 工作。

目标：
<明确本阶段目标，例如：根据用户访谈结果生成冻结前需求文档草案>

必须先读取：
- AI指导.md
- AI项目构建流程.md
- <当前阶段相关流程文档>
- <已有需求/接口/架构文档>

约束：
- 不跳过 Grill Gate。
- 不在需求未冻结前编写业务代码。
- 不把扩展点实现为持续增长的 if/else 或 switch/case。
- 所有模糊项必须进入待确认清单。
- 所有真实密钥、Token、密码和连接串不得写入代码、文档、测试或日志。

完成条件：
- 输出目标阶段的文档草案或实现结果。
- 列出已明确内容、待确认问题、决策记录、扩展点清单和验收标准。
- 明确说明是否可以进入下一阶段。
- 如果存在 P0/P1 模糊项，必须停止并请求用户裁决。

迭代策略：
每完成一个检查点，先自查是否满足对应流程文档的退出标准，再继续下一步。

阻塞条件：
如果缺少关键输入、出现冲突、无法验证完成条件，停止并报告需要用户补充什么。
```

## 4. 阶段准入规则

### 4.1 进入需求文档阶段

必须读取：

- `AI指导.md`
- `需求文档设计流程.md`

必须完成：

- Grill Gate。
- 项目类型识别。
- MVP 与 Out of Scope 初判。
- 扩展点初判。

### 4.2 进入接口文档阶段

必须读取：

- `AI指导.md`
- `接口文档设计流程.md`
- 冻结后的需求资产。

必须完成：

- 需求冻结退出检查。
- 针对接口契约的 Grill-With-Docs。
- E2+ 扩展点契约映射。

### 4.3 进入项目初始化阶段

必须读取：

- `AI指导.md`
- `AI项目构建流程.md`
- `最优项目初始化结构.md`
- 冻结后的需求资产。
- 冻结后的接口契约。

必须完成：

- Goal Gate。
- 项目执行级别选择。
- `MODULE_MAP.md` 扩展点归属、注册表路径和禁止修改核心流程的定义。

### 4.4 进入代码构建阶段

必须读取：

- `AI指导.md`
- `AI_AGENT_CONTRACT.md`
- `agent.md`
- `MODULE_MAP.md`
- 当前模块 `src/<module_name>/agents.md`
- 当前模块 `src/<module_name>/tasks.md`
- 当前模块相关需求和契约文档。

必须完成：

- 只修改允许路径。
- 按扩展强度选择对应实现模式。
- 编写或更新验证。
- 运行 Hook 或等价验证。
- 更新对应 `tasks.md`。

## 5. 统一输出格式

每个阶段完成后，AI 必须输出：

```md
## 阶段结果
- 当前阶段：
- 是否通过：
- 下一阶段：

## 已明确内容
- ...

## 待确认问题
无待确认问题时，保留表头并填 `N/A`，不要编造占位数据。

| 编号 | 问题 | 阻塞等级 | 影响 | 建议处理 |
|---|---|---|---|---|

## 扩展点清单
无扩展点时，保留表头并填 `N/A`，不要编造占位数据。

| 扩展点 | 强度 | 推荐模式 | 归属模块 | 新增成员方式 | 禁止修改 |
|---|---|---|---|---|---|

## 决策记录
- ...

## 验证与退出标准
- 已检查：
- 未满足：
```

## 6. 禁止行为

- 禁止未完成 Grill Gate 就生成最终 PRD。
- 禁止未完成需求冻结就生成物理接口或业务代码。
- 禁止未完成接口冻结就构建模块代码。
- 禁止把持续增长的扩展点写成连续 `if/else` 或 `switch/case`。
- 禁止将真实密钥、Token、密码、连接串写入代码、文档、测试、脚本或日志。
- 禁止只改代码不更新需求、接口、MODULE_MAP、Agent 上下文或测试。
- 禁止使用“可能、应该、后续再说、视情况”等模糊词作为冻结结论。

## 7. 工具能力探测与自适应路径 (Tool Capabilities Checklist)

为了能在不同的 AI 辅助环境中取得最佳的自动化效果，AI 进入本项目时应主动执行以下环境自检，并自适应切换到最契合的执行模式：

### 7.1 工具能力探测列表

1. **斜杠命令支持度**：
   - **检查方式**：检查当前编辑器或聊天框是否支持 `/grill-me`、`/grill-me-doc` 或 `/goal` 等快捷命令。
   - **自适应**：
     - *支持*：直接运行对应斜杠命令启动网关。
     - *不支持*：自动 fallback 到本指南第 2.3 和 3.3 节的**兼容提示词模式**，严格走问答澄清和目标锁定流程。
2. **多 Agent 派发与生命周期管理**：
   - **检查方式**：检查平台或 IDE 是否具有“派发子任务/启动子 Agent”的能力（如 `invoke_subagent` 工具、MCP 跨实例通信或支持 `agent_prompts/` 初始化）。
   - **自适应**：
     - *支持*：由 OrchestratorAgent 运行流水线总调度，派发独立的有状态 ModuleAgent 进行代码开发，并启动只读的 ReviewAgent 输出审查报告。
     - *不支持*：退化为**单会话角色模拟模式**，在同一个会话中通过切换系统提示词依次完成“总控 -> 编码 -> 审查”三角色闭环，确保隔离性。
3. **本地 Hook 自动/手动触发机制**：
   - **检查方式**：检查工具是否拥有本地终端命令执行权限（如 `run_command`），以及 Git 是否挂载了 Hook。
   - **自适应**：
     - *支持*：代码变更后，AI 主动在终端静默运行 `python hooks/lite_hook.py` 或 `python hooks/finalize.py`，根据执行返回的 JSON 契约报告判定是否可进入下一流程。
     - *不支持*：停止操作，显式引导用户在本地运行 `python hooks/finalize.py`，并请求用户将输出的 JSON 报告黏贴回对话中，以此作为继续运行的“证据约束”。
