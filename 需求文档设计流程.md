# 高质量需求文档设计流程通用规范

> 适用场景：在项目构建前，使用 AI 工具引导用户明确需求、生成 PRD、拆分模块、形成逻辑实体与事件流草案，并通过回归检查和盲审把需求冻结为可构建资产。

需求文档是整个项目的第一源头。后续接口文档、Agent 分工、模块构建、测试验收和人工检查，都必须从需求文档追溯而来。

本流程不默认项目一定是 Web 应用。需求采集必须先识别项目类型，再按 Web、App、CLI、插件、API 服务、数据任务、自动化脚本、桌面端、AI Agent、游戏等不同形态分支追问。

> **定位说明**：本文是 `AI项目构建流程.md` 的 ReqOps 阶段细化规范，不单独作为项目全流程入口。项目执行级别、裁剪方式和 Hook 约束以主构建流程为准。

> **入口补充**：如果当前会话使用 `/grill-me`、`/grill-me-doc` 或 `/goal`，AI 必须先执行这些目标对齐动作，再进入本流程的需求采集与文档生成。

## 1. 核心原则

- 需求文档必须先于接口文档和代码构建。
- 需求文档生成前必须先通过 Grill Gate；AI 必须完成一问一答式澄清，不能跳过模糊项直接生成最终 PRD。
- 模糊问题不能进入冻结版 PRD，必须追问、标记待确认，或由人工裁决。
- AI 不能自行假设用户没有明确说明的关键业务规则、技术栈、权限边界、数据口径或验收标准。
- MVP 范围和 Out of Scope 必须同时确认。
- AI 必须主动识别未来可能扩展的业务维度，例如平台、渠道、内容类型、支付方式、通知方式、第三方服务、角色权限、导入导出格式、AI 工具、策略规则等；不得把这些变化维度隐藏在一组不断增长的 `if/else` 需求描述中。
- 启动问题清单是渐进式问题库，不是一次性表单。
- 术语表在 PRD 编写期间保持增量更新，最终冻结必须与 PRD 冻结同步完成。
- 需求阶段只输出逻辑实体、系统边界和事件流草案；物理 API 协议、路径、字段细节进入构建流程后再定稿。
- 哈希、追踪矩阵生成等机械工作必须由 Hook 或工具自动完成，不要求人工手动维护。
- 每次需求变更都必须记录影响范围，并重新触发回归检查。

## 2. 总体流程与 AI 交互行为

当用户在项目之初“丢出”本需求文档设计流程与初始构想时，AI **不得直接生成最终 PRD**。必须遵循以下交互闭环：

```text
1. AI 读取初始构想，扫描并识别项目类型候选
   ↓
2. 【Grill Gate】AI 梳理出所有模糊、不确定或冲突的业务逻辑，列出“模糊与待确认问题清单”，并一次只追问一个问题
   ↓
3. 渐进式问题采集与用户交互澄清（Intake-1 ~ Intake-3 渐进式沟通，禁止一次性抛出百条表单）
   ↓
4. 生成《初始需求信息包》，明确 MVP 范围与 Out of Scope，初步判断 E0-E4 扩展点
   ↓
5. 冻结核心不可变原则（constitution.md），同步初始化业务术语表（glossary.md）
   ↓
6. 基于已澄清信息，生成《总览 PRD 初稿》（概述、核心流程、状态机、数据对象、验收标准等）
   ↓
7. 拆分模块需求，完善 MODULE_MAP，同步输出逻辑实体与事件流草案（interfaces/）
   ↓
8. 触发 CriticAgent 批判、RegressionAgent 范围检查与 Auditor 盲审（视项目级别裁剪）
   ↓
9. 最多 3 轮“审查-修订”闭环
   ↓
10. Human Owner 确认并同步冻结全部需求资产包（移交给接口设计流程）
```

## 3. RequirementGuideAgent 提示词

下面这段提示词可以交给任意 AI，用于初始化“需求文档引导者”角色。

```md
# RequirementGuideAgent

你是一个高质量需求文档引导者。你的任务不是直接写代码，也不是立刻生成最终 PRD，而是通过结构化访谈帮助用户把项目需求说清楚。

## 目标

你需要引导用户明确以下内容：

- 项目类型、项目目标和核心用户。
- MVP 范围和 Out of Scope。
- 核心业务流程、状态机、数据对象。
- 项目类型对应的技术、运行环境、交付方式和外部依赖。
- 权限、安全、隐私、性能、日志、异常处理等非功能要求。
- 模块边界、逻辑实体、事件流和验收标准。
- 未来扩展维度、变化频率、扩展强度候选，以及哪些扩展必须通过策略、工厂、适配器、插件或事件机制承载。

## 严格规则

- 任何模糊、不确定、互相冲突的地方，必须提出问题。
- 不允许把“可能、应该、视情况、适当、后续再说”等模糊描述写入冻结版需求。
- 用户没有明确回答的关键项，必须标记为 `待确认`。
- 如果用户表示“不确定”，你必须给出 2 到 3 个可选方案，并说明取舍，再让用户确认。
- 不得自行决定业务规则、权限边界、数据口径、技术栈或验收标准。
- 不得默认项目是 Web 应用；必须先确认项目类型。
- 不得把平台、渠道、供应商、内容类型、支付方式、模型工具等变化维度写成“以后在代码里加判断”的需求；必须把它们标记为扩展点，并说明新增成员时不应修改哪些核心流程。
- 每轮沟通后，必须输出：
  - 已明确内容
  - 待确认问题
  - 冲突或风险
  - 建议下一步

## 工作流程

1. 先使用“渐进式问题采集”识别项目类型和核心业务。
2. 将用户回答整理成“初始需求信息包”。
3. 根据项目类型选择 Web、App、CLI、插件、API 服务、数据任务、自动化脚本、桌面端等分支问题。
4. 对缺失、模糊、冲突的地方继续追问。
5. 对未来扩展维度进行单独追问，输出扩展点清单、扩展强度候选和推荐设计模式。
6. 在信息足够后，生成核心不可变原则、术语表初版和 PRD 初稿。
7. 在 PRD 编写期间增量维护术语表。
8. 生成模块需求、逻辑实体/事件流草案、扩展点草案和验收标准。
9. 触发只读审查；高风险或 Enterprise 项目再启用 RegressionAgent 与 Auditor。
10. 只有在无 P0/P1 模糊项或缺陷时，才能建议进入需求冻结。
```

## 4. 渐进式问题采集

本节是问题库，不是一次性表单。AI 不应要求使用者一口气回答所有问题，而应先收集核心信息，再根据项目类型、模块边界和 PRD 初稿中的模糊点进行定向追问。

### 4.1 推荐采集节奏

| 轮次 | 目标 | 问题范围 | 输出 |
|---|---|---|---|
| Intake-1 | 识别项目类型和核心业务 | Q1-Q20 | 粗颗粒度项目框架 |
| Intake-2 | 建立核心流程、数据和验收标准 | Q21-Q34、Q70-Q75 | 初始需求信息包 |
| Intake-3 | 按项目类型分支追问 | 只选择相关类型的问题 | 技术与交付约束 |
| Intake-4 | 根据 PRD 初稿定向追问 | 模块级模糊项 | 可审查 PRD 初稿 |

### 4.2 项目类型分流

| 项目类型 | 后续重点追问 |
|---|---|
| Web / 管理后台 / SaaS | 页面、组件库、前端状态、后端 API、权限、部署 |
| 移动 App / 小程序 | 端能力、离线、推送、审核、设备权限 |
| CLI / 桌面工具 | 命令、参数、文件输入输出、本地环境、打包发布 |
| 浏览器插件 | 插件权限、内容脚本、后台脚本、宿主页交互、商店发布 |
| 服务端 API / SDK | 协议、鉴权、限流、版本、调用方、错误模型 |
| 数据任务 / ETL / 报表 | 数据源、转换规则、调度、口径、导出格式 |
| 自动化脚本 / 运维工具 | 运行环境、权限、幂等、日志、失败回滚 |
| AI Agent / 工作流 | 工具权限、记忆、上下文、循环上限、人工确认点 |
| 游戏 / 互动体验 | 规则、关卡、资产、交互、实时性、平台 |

如果用户无法判断项目类型，RequirementGuideAgent 必须给出 2 到 3 个最可能的类型选项，并说明每种类型对后续需求文档的影响。

### 4.3 核心问题

| 编号 | 问题 | 目的 |
|---|---|---|
| Q1 | 项目名称是什么？是否已有代号或正式名称？ | 确认文档和模块命名 |
| Q2 | 用一句话描述这个项目要解决什么问题？ | 提炼项目目标 |
| Q3 | 为什么现在需要做这个项目？ | 确认业务背景 |
| Q4 | 项目的主要使用者是谁？ | 明确用户类型 |
| Q5 | 项目最终交付形态是什么？Web、App、桌面端、CLI、插件、服务端 API、数据任务，还是组合？ | 确认项目类型 |
| Q6 | 这是新项目、旧项目重构，还是现有系统加功能？ | 判断约束来源 |
| Q7 | 是否有竞品、参考产品或现有流程可以对照？ | 获取设计参照 |
| Q8 | 项目成功的最重要标准是什么？ | 明确高层 KPI |
| Q9 | 有哪些用户角色？ | 建立角色模型 |
| Q10 | 每类用户最核心的使用场景是什么？ | 明确核心路径 |
| Q11 | 第一版必须完成哪些功能？ | 定义 MVP |
| Q12 | 哪些功能可以延期？ | 控制范围 |
| Q13 | 哪些功能明确不做？ | 形成 Out of Scope |
| Q14 | 哪些功能如果不做，项目就没有价值？ | 找出核心能力 |
| Q15 | 用户从开始到完成目标的完整步骤是什么？ | 梳理主流程 |
| Q16 | 流程中有哪些状态？ | 建立状态机 |
| Q17 | 哪些步骤可能失败？失败后如何恢复？ | 明确异常路径 |
| Q18 | 系统中最核心的数据对象有哪些？ | 建立数据模型 |
| Q19 | 是否存在敏感数据、个人信息、财务数据或业务机密？ | 明确安全要求 |
| Q20 | 每个核心功能如何判断完成？ | 形成验收标准 |
| Q31 | 未来最可能新增哪些类别？例如平台、渠道、内容类型、供应商、支付方式、导入导出格式、AI 工具或业务规则。 | 识别扩展维度 |
| Q32 | 每个扩展维度的变化频率如何？一次性、偶尔新增、持续新增，还是由第三方生态驱动？ | 判断扩展强度 |
| Q33 | 新增一个扩展成员时，期望只改配置、只新增适配器文件，还是允许修改核心流程？ | 明确扩展边界 |
| Q34 | 不同扩展成员之间有哪些共同行为和差异行为？例如发布、校验、鉴权、限流、回调、状态同步、错误恢复。 | 抽象能力模型 |
| Q35 | 是否需要让第三方或后续团队独立接入扩展？是否需要插件机制、注册表、能力声明或版本兼容策略？ | 判断是否需要插件化 |
| Q36 | 哪些业务规则看起来像条件分支，但本质上应抽象为策略、工厂、适配器、状态机或规则引擎？ | 防止分支堆积 |

### 4.4 类型分支问题

按项目类型选择相关问题，不相关的类型问题不强制回答。

| 类型 | 必问方向 |
|---|---|
| Web / 管理后台 | 前端框架、组件库、页面清单、响应式、浏览器兼容、后端服务、登录权限 |
| 移动 App / 小程序 | 端平台、审核规则、推送、离线、设备权限、账号体系、版本发布 |
| CLI / 桌面工具 | 命令模型、参数、配置文件、输入输出、系统兼容、安装升级 |
| 浏览器插件 | 权限声明、内容脚本、后台脚本、宿主页交互、真实发布与商店规则 |
| 服务端 API / SDK | 调用方、协议偏好、鉴权、错误模型、限流、版本、兼容性 |
| 数据任务 / 报表 | 数据源、清洗规则、调度频率、指标口径、导出格式、失败重跑 |
| 自动化脚本 | 运行环境、权限、幂等、日志、回滚、安全边界 |
| AI Agent / 工作流 | 工具权限、上下文来源、循环轮数、人工确认点、失败兜底 |
| 游戏 / 互动体验 | 玩法规则、胜负条件、关卡、资产、输入方式、目标平台 |

|---|---|---|

| Q22 | 响应时间、处理时长或资源消耗有什么要求？ | 明确性能 |
| Q23 | 是否需要日志、指标、告警、审计或链路追踪？ | 明确可观测性 |
| Q24 | 是否需要角色权限、操作日志、审批或风控？ | 明确安全与合规 |
| Q25 | 是否有部署环境限制？本地、服务器、Docker、云平台、内网。 | 明确部署 |
| Q26 | 是否需要自动化测试、CI/CD 或发布审批？ | 明确工程要求 |
| Q27 | 是否使用多个 AI 工具协作？分别负责什么？ | 明确 AI 分工 |

| Q30 | 谁负责最终验收？ | 明确验收责任 |

### 4.6 扩展性专项追问

当项目出现“多平台、多渠道、多供应商、多内容类型、多支付方式、多模型、多工具、多策略、多格式、多角色差异”等信号时，RequirementGuideAgent 必须启动扩展性专项追问。典型问题包括：

| 扩展场景 | 必须追问 |
|---|---|
| 多平台发布 | 目标平台清单、每个平台的能力差异、鉴权方式、内容格式、审核规则、发布状态、失败重试、回调或同步机制 |
| 多内容类型 | 内容类型清单、共同字段、差异字段、校验规则、渲染或发布策略、生命周期状态 |
| 多第三方服务 | 服务商差异、凭据管理、限流规则、错误码映射、重试策略、降级策略、替换成本 |
| 多支付或交易方式 | 支付渠道、结算状态、幂等键、退款/撤销、资金风险、审计要求 |
| 多导入导出格式 | 格式清单、Schema、字段映射、版本兼容、错误报告、批处理失败恢复 |
| 多 AI 工具或模型 | 工具能力声明、输入输出 Schema、权限边界、失败回退、人工确认点 |
| 多业务策略 | 策略输入、输出、选择条件、默认策略、优先级、冲突处理、是否需要规则引擎 |

扩展性专项追问的目标不是预先实现所有未来功能，而是明确：

- 哪些变化维度属于 MVP 必须支持的扩展点。
- 哪些变化维度只需要在架构上预留，不进入第一版功能。
- 新增扩展成员时，允许修改哪些文件，不允许修改哪些核心流程。
- 扩展成员是否需要统一能力声明、注册表、适配器、策略对象、插件接口或事件订阅机制。
- 哪些场景可以接受少量条件判断，哪些场景必须禁止继续堆叠 `if/else` 或 `switch/case`。

### 4.7 扩展强度分级

需求阶段必须为每个扩展点标记强度，供接口文档和构建流程选择对应实现模式。

| 强度 | 适用情况 | 需求阶段输出 |
|---|---|---|
| E0 无需抽象 | 只有 1 个实现，且明确不会扩展 | 记录不抽象原因，禁止过度设计 |
| E1 轻量枚举 | 成员少、变化低、差异只影响展示或简单文案 | 枚举值、差异字段、验收场景 |
| E2 策略/适配器 | 成员会增加，且差异影响校验、流程、外部调用或错误处理 | 能力接口、选择规则、默认策略、扩展示例 |
| E3 插件化/注册表 | 第三方或后续团队会独立新增成员，或成员数量持续增长 | 注册表、能力声明、版本兼容、隔离边界、审核流程 |
| E4 规则引擎/工作流 | 规则组合复杂、频繁调整、需要非开发人员配置 | 规则输入输出、优先级、冲突处理、审计和回滚 |

卡点：

- 任一扩展维度达到 E2 及以上时，PRD 必须包含扩展点清单。
- 任一扩展维度达到 E3 及以上时，接口文档必须定义注册表或插件契约。
- 任一扩展维度达到 E4 时，必须由 Human Owner 裁决是否引入规则引擎或工作流引擎，不能由 AI 默认选择。

## 7. 需求资产职责

### `raw-intake.md`

保存渐进式问题采集的原始回答、AI 整理后的初始需求信息包，以及所有待确认问题。

### `constitution.md`

核心不可变原则。用于约束后续 PRD、模块需求、逻辑实体、事件流和代码构建。

### `glossary.md`

统一业务术语表。术语表不是在 PRD 之前一次性冻结的文件，而是在 PRD 编写期间持续增量维护，并在需求冻结时与 PRD 一起冻结。

### `REQUIREMENT_CONTRACT.md`

需求阶段薄契约文件。只记录冻结需求资产的读取顺序、冻结版本、必备交付物、进入接口阶段的准入条件和阻塞条件；不复制 PRD 正文、不承载动态任务，也不替代 `AI_AGENT_CONTRACT.md` 的全局执行规则。

### `overview.md`

最终 PRD 总览文档。用于描述项目目标、用户、场景、MVP、流程、状态机、数据对象、非功能需求和验收标准。

### `out-of-scope.md`

明确不做什么。RegressionAgent 必须基于该文件执行范围回归检查。

### `interfaces/logical-model.md`

逻辑实体模型。只描述业务实体、关键属性、关系和约束，不定义 HTTP 路径、GraphQL Schema、数据库字段或 SDK 方法签名。

### `interfaces/events.md`

逻辑事件流。只描述业务事件、触发方、接收方、状态变化和负载语义，不绑定具体传输协议。

### `traceability-source.yaml`

需求追踪矩阵的结构化源文件。禁止人工维护 CSV。该文件由 OrchestratorAgent 或 Hook 根据 Markdown 文档注解生成或更新。

### `traceability.generated.md`

由 `traceability-source.yaml` 自动生成的人类可读追踪矩阵。

## 8. Agent 角色分工

| Agent | 职责 | 是否可改文档 |
|---|---|---|
| RequirementGuideAgent | 通过渐进式问题采集引导用户明确初始需求 | 可以 |
| RequirementAgent | 生成和修订 PRD、模块需求、逻辑实体与事件流草案 | 可以 |
| Human Owner | 裁决业务边界、优先级、技术取舍和冻结版本 | 可以裁决 |
| GlossaryAgent | 提取并增量维护统一业务术语表；Lite/Standard 可由 RequirementAgent 兼任 | 可以 |
| CriticAgent | 从 QA、数据、SRE、安全、业务等视角提出缺陷；Standard 按风险启用，Enterprise 默认启用 | 不建议直接修改 |
| RegressionAgent | 检查 Out of Scope、验收标准和核心原则是否被破坏；范围漂移或 Enterprise 默认启用 | 不建议直接修改 |
| Auditor | 异构模型盲审，输出量化评分、Gap List 和 PASS/REJECT；Enterprise 默认启用，Standard 仅高风险启用 | 不可以 |
| OrchestratorAgent | 维护模块边界并把冻结需求移交构建流程 | 可以 |

裁剪规则：

- Lite 只需要 `RequirementGuideAgent`、`RequirementAgent` 和 `Human Owner`，不得强制引入盲审。
- Standard 默认三角色闭环：需求生成者、只读审查者、Human Owner；只在需求漂移、高风险数据、安全隐私或反复冲突时启用 `RegressionAgent` / `Auditor`。
- Enterprise 启用完整角色分工，并要求审查输出可追溯证据。

## 9. 核心不可变原则冻结

`constitution.md` 必须包含标准化元数据，但 `sha256_hash` 禁止人工填写。

```yaml
constitution_metadata:
  constitution_version: "v1.0.0"
  sha256_hash: "<AUTO_GENERATED_BY_HOOK>"
  signer: "<Human Owner>"
  signed_at: "<ISO-8601 time>"
  source_input_version: "Raw_Req_v1.0"
  change_reason: "项目首次初始化"
```

Hash 规则：

- 人类只维护语义版本号、签署人、签署时间和变更原因。
- `sha256_hash` 必须由 Hook、脚本或集成工具自动计算并填充。
- 如果没有自动化工具，不启用 Hash 卡点，只使用语义版本号和 Decision Log。
- 格式微调不得触发人工重算 Hash。

卡点：

- 未签署核心不可变原则，不进入 PRD 冻结。
- 若核心原则需要修改，禁止直接覆盖原文，必须走 Amendment Flow。

## 13. 多角色批判与修订

CriticAgent 可以按视角拆分：QA、Data、SRE、Security、Business。

严重级别定义：

| 级别 | 定义 | 是否阻塞冻结 |
|---|---|---|
| P0 | 破坏核心目标、KPI、资金/隐私安全或主流程不可闭环 | 是 |
| P1 | 核心功能不可验收、模块边界冲突、逻辑实体或事件流无法承载状态机 | 是 |
| P2 | 重要异常路径、非功能要求或测试标准不完整 | 视情况 |
| P3 | 表述、格式、示例、低风险优化建议 | 否 |

Resolution Log 和 Decision Log 必须记录修订原因、取舍和影响范围。

## 14. RegressionAgent 回归检查

RegressionAgent 不提新建议，只做破坏性自测。

断言规则：

1. 是否偷偷加回 Out of Scope 范围。
2. 是否导致已有验收标准失效。
3. 是否改变核心不可变原则中的 KPI 统计口径。
4. 是否让逻辑实体或事件流无法承载业务状态机。
5. 是否改变已签署的权限、安全或隐私边界。

存在 P0/P1 回归时，不进入 Auditor 盲审。

## 15. Auditor 异构盲审

Auditor 必须只读审查，不修改文件。

评分维度：

| 维度 | 说明 | 通过阈值 |
|---|---|---|
| constitution_alignment_score | 是否遵循核心不可变原则和优先级 | >= 8 |
| technical_feasibility_score | 结合项目类型、技术候选、团队能力和工期是否可落地 | >= 8 |
| business_closure_score | 状态机、异常流、逻辑实体和事件流是否无逻辑断层 | >= 8 |
| acceptance_testability_score | 验收标准是否能直接转成测试用例 | >= 8 |
| risk_control_score | 合规、隐私、安全、资损、失败兜底是否充分 | >= 7 |

Auditor 给出 REJECT 时，必须输出 Gap List：

```md
## Gap List

| Gap | Score Impact | Blocking Level | Evidence | Required Fix | Owner |
|---|---|---|---|---|---|
| <缺口> | <扣分维度> | P0/P1/P2/P3 | <证据> | <需要补齐什么> | <负责人> |
```

证据规则：

- 每个低于阈值的评分必须绑定至少一条 Gap List。
- `Evidence` 必须包含具体文档路径、章节标题或需求编号；不能只写“描述不清”“边界不足”等泛化判断。
- `Required Fix` 必须说明要补齐的内容和验证方式，不能只写“优化”“完善”“进一步确认”。
- 没有证据的低分只能作为观察项，不能作为 REJECT 的阻塞依据。

三态结论：

- PASS：所有分数达到阈值，且无 P0/P1。
- CONDITIONAL PASS：无 P0/P1，但存在资源、技术栈、延期风险或 Human Owner 明确接受的未达标项。
- REJECT：任意关键分数低于 6，或存在未解决 P0/P1。

Owner 覆盖规则：

- 第 3 轮仍未通过时，Human Owner 可以通过 Decision Log 将特定未达标项移入 Pending Log。
- 被覆盖的问题必须不是 P0 安全、隐私、资损或主流程闭环问题。
- 覆盖后状态只能是 CONDITIONAL PASS，不能伪装为 PASS。

## 17. 变更治理

需求冻结后仍可变更，但必须走 Amendment Flow。

流程：

1. 暂停受影响的构建 Agent。
2. 填写需求修正案。
3. 生成新语义版本号。
4. Hook 自动刷新 `sha256_hash`。
5. Human Owner 重新签署。
6. 刷新全局上下文。
7. 只读视角重新检查；高风险或 Enterprise 项目启用 RegressionAgent。
8. 高风险或 Enterprise 项目由 Auditor 对变更部分盲审。
9. OrchestratorAgent 同步给构建流程。

## 19. 需求冻结退出标准

需求文档可以进入构建流程，需要同时满足：

- 渐进式问题采集已完成，关键问题无 P0/P1 模糊项。
- Grill Gate 已完成；关键追问、推荐答案、用户裁决和未解决问题已记录。
- 项目类型已明确，类型相关约束已记录。
- 初始需求信息包已生成。
- 核心不可变原则已签署。
- 统一业务术语表已与 PRD 同步冻结。
- MVP 和 Out of Scope 已明确。
- 总览 PRD 已完成。
- 模块需求已完成。
- 逻辑实体和事件流已覆盖核心路径和状态机。
- 扩展点清单、扩展强度、推荐模式、新增成员允许修改和禁止修改的边界已冻结；E2 及以上扩展点无 P0/P1 待确认项。
- 追踪矩阵源文件和生成文件已更新。
- 只读审查无 P0/P1 回归问题；若启用 RegressionAgent，则其报告无 P0/P1。
- 若启用 Auditor，则 Auditor 无 P0/P1 文档问题。
- 若启用 Auditor，`business_closure_score` 达到阈值，或 Human Owner 通过 Decision Log 授权 CONDITIONAL PASS。
- Human Owner 确认可以移交构建流程。

## 20. 最小使用方式

如果不想一次执行完整 ReqOps 流程，至少执行以下最小闭环：

1. 使用渐进式问题采集识别项目类型和核心业务。
2. 生成初始需求信息包。
3. 追问所有 P0/P1 模糊项。
4. 生成核心不可变原则、术语表初版和 Out of Scope。
5. 生成 PRD、模块需求、逻辑实体和事件流。
6. 由一个只读视角检查 Out of Scope、验收标准和状态机闭环。
7. 如果项目达到 Enterprise 或存在安全、隐私、资金、合规风险，再执行 Auditor 盲审并输出 Gap List。
